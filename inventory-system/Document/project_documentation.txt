## 🛠️ SECTION 1: SYSTEM END-TO-END TROUBLESHOOTING LOG
This log details the exact architectural, procedural, and environmental roadblocks discovered during implementation and how they were systematically resolved.
## Issue 1: HTTP Status 404 – Not Found on Node Initialisation

* Symptom: Opening http://localhost:8092/inventory-system/ threw a clean 404 status page from Tomcat.
* Root Cause Investigation: Checked the physical file explorer under ...\tomcat1\Tomcat 9.0\webapps\. The source inventory-system.war file was present, but Tomcat had not unzipped or auto-deployed it into a matching application folder. The instance was either stopped or frozen.
* In-Depth Solution Step-by-Step:
1. Forcibly exited all running backend process boxes.
   2. Cleared out the locked, un-extracted .war remnants from the target folder directory using terminal filesystems.
   3. Re-triggered the dedicated instance runner script manually via isolated terminal tabs.
   4. Monitored the webapps folder for 5 seconds until the yellow extraction folder natively appeared, completing auto-deployment.

## Issue 2: Terminal Space Parsing Error (CommandNotFoundException)

* Symptom: Executing direct absolute string paths caused severe PowerShell script translation failures.
* Root Cause Investigation: The target installation folders contained whitespace characters within their names (...\tomcat1\Tomcat 9.0\...). PowerShell interpreted everything following the space as an unrecognised separate command line argument.
* In-Depth Solution Step-by-Step:
1. Bound the absolute file paths tightly using double quotation marks (" ").
   2. Appended the native execution operator token ampersand (&) at the beginning of the path string line to instruct PowerShell to cleanly run the full wrapped path string as a single unified binary application file.

## Issue 3: CATALINA_HOME Environment Variable Misalignment

* Symptom: Running the startup scripts threw the block error: "The CATALINA_HOME environment variable is not defined correctly".
* Root Cause Investigation: Because multiple separate Tomcat installations were being called sequentially within the same terminal operational space, the shell environment profile locked its routing pointers to the very first folder instance it executed. This blocked subsequent nodes from initializing.
* In-Depth Solution Step-by-Step:
1. Abandoned using a single monolithic shell window context.
   2. Split the workflow into 3 independent, isolated terminal tabs inside VS Code (named Tomcat 1, Tomcat 2, Tomcat 3).
   3. Before running any script, explicitly declared isolated environment variables ($env:CATALINA_HOME and $env:CATALINA_BASE) directly at the top of their respective tabs to force-feed Tomcat the exact folder location it belonged to.

## Issue 4: Nginx Crash on Initial Launch (Socket Binder Exception 10013)

* Symptom: Executing .\nginx.exe threw a fatal runtime error: bind() to 0.0.0.0:80 failed (10013: An attempt was made to access a socket...).
* Root Cause Investigation: Nginx defaults to listening on port 80. On standard consumer Windows editions, port 80 is often permanently hijacked by default native background OS operations (such as World Wide Web Publishing Services or Microsoft IIS).
* In-Depth Solution Step-by-Step:
1. Opened conf/nginx.conf inside the VS Code editor.
   2. Navigated to line 14 within the server descriptor matrix block.
   3. Changed the entry listen 80; to an alternative unassigned port number (listen 8000;). This successfully bypassed the network access control lock.

## Issue 5: 502 Bad Gateway Output

* Symptom: Browsing http://localhost:8000/inventory-system/ loaded an Nginx branded 502 Bad Gateway screen.
* Root Cause Investigation: The load balancer was active on port 8000, but it couldn't talk to the backend. The nginx.conf file was using the string word localhost in its upstream configuration block, causing Windows DNS lookup looping delays, or the backend Tomcat instances had silently shut down.
* In-Depth Solution Step-by-Step:
1. Re-verified all 3 Tomcat instances were awake by testing their separate individual direct browser links (8091, 8092, 8093).
   2. Replaced the generic localhost text string inside the upstream proxy pool with explicit loopback numerical IP notation addresses (127.0.0.1).
   3. Executed a complete background pipeline termination (taskkill /f /im nginx.exe) and started the proxy load balancer engine fresh.

## Issue 6: Static Page Layout Text Freeze (No Port Change on Refresh)

* Symptom: The proxy cluster was functional, but clicking refresh kept displaying the text phrase CLUSTER NODE: DEFAULT SERVER without showing any port changes.
* Root Cause Investigation: The initial UI layout configuration used a fixed text string rather than calling the dynamic server runtime script. Additionally, aggressive browser session cache retention prevented the tab from loading fresh routing decisions.
* In-Depth Solution Step-by-Step:
1. Edited dashboard.jsp line 65, stripping out the static placeholder characters.
   2. Implemented the dynamic Java expression tag script element <%= request.getLocalPort() %>.
   3. Ran a full Maven re-compilation (mvn clean package) and distributed the fresh WAR file across all nodes.
   4. Executed the final browser validation using an Incognito Window (Ctrl + Shift + N) to force clean round-robin asset rendering.

## 💻 SECTION 2: COMMAND REFERENCE & SYSTEM USE CASES
This section contains descriptions of every single terminal instruction utilized throughout the DevOps project pipeline.
## 1. Project Scaffolding

* Command: mvn archetype:generate "-DgroupId=com.inventory" "-DartifactId=inventory-system" "-DarchetypeArtifactId=maven-archetype-webapp" "-DinteractiveMode=false"
* Technical Explanation: Invokes the Maven Archetype plugin to construct a standardized directory skeleton based on the default web application structural boilerplate archetype.
* Project Use Case: Executed inside the base project folder to generate a clean web application directory layout (src/main/webapp/WEB-INF/) automatically, preventing structural configuration errors.

## 2. Compilation and Artifact Packaging

* Command: mvn clean package
* Technical Explanation: A multi-phase Maven command. clean wipes out any old compiled file artifacts in the /target directory. package compiles the source Java classes, executes validation checks, and bundles the entire web application assets into a deployable package format.
* Project Use Case: Executed right inside the project root folder whenever any changes were made to the Java files or JSP code, ensuring the system generated a fresh, working inventory-system.war file.

## 3. Folder Automation Management

* Command: mkdir -p src/main/java/com/inventory/controller
* Technical Explanation: The directory creation command. The flag -p acts as an automated validation switch that creates all missing parent folders across the defined path at once without throwing errors if they exist.
* Project Use Case: Run immediately after generating the archetype to safely insert the missing standard Java compilation source package paths into the project hierarchy.

## 4. File System Extraction Verification

* Command: Copy-Item -Path "source" -Destination "target" -Force
* Technical Explanation: A native PowerShell data management command that moves file assets across filesystem locations. The -Force parameter overrides read-only asset permissions or locks.
* Project Use Case: Used to copy the newly compiled inventory-system.war file into the separate Tomcat webapps/ folders to prepare them for deployment.

## 5. Infrastructure Runtime Execution

* Command: & "path\startup.bat"
* Technical Explanation: The PowerShell execution operator. It tells the terminal session to treat the quoted string literal path as an active application binary file and run it directly.
* Project Use Case: Run inside the respective terminal instances to launch the background Tomcat operational scripts safely, even if there are blank spaces in the path layout.

## 6. Process Level Interception & Management

* Command: taskkill /f /im nginx.exe
* Technical Explanation: A Windows terminal utility instruction that kills running applications. The switch /f forces the immediate termination of the process, and /im targets the application using its image name.
* Project Use Case: Used to completely shut down frozen or hidden background instances of Nginx from system memory when modifying configuration routing maps.

       upstream inventory_cluster {
