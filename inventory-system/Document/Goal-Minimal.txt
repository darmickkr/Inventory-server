## 🧠 Core Concepts Implemented## 1. Horizontal Scaling vs. Vertical Scaling

* Vertical Scaling means adding more power (RAM, CPU) to a single machine. It has limits and requires downtime.
* Horizontal Scaling (What you did!) means adding more machines or server instances to share the workload. This allows for infinite growth without relying on a single computer. [7, 8, 9, 10, 11] 

## 2. High Availability & Fault Tolerance

* High Availability ensures your website is accessible almost 100% of the time.
* Fault Tolerance means if you go to your desktop right now and close the tomcat1 window, your application will still work perfectly because Nginx will instantly redirect traffic to tomcat2 and tomcat3 without the user noticing.

## 3. Reverse Proxy & Infrastructure Abstraction

* Instead of making users remember messy ports like :8091 or :8092, Nginx acts as a Reverse Proxy.
* It hides your backend servers entirely. The user only ever sees http://localhost/inventory-system/. [12]

## ⚙️ Technical Terms You Used

* WAR (Web Application Archive): A standard compressed file layout containing your compiled Java classes, HTML, JSP files, and web configurations, packaged specifically to run inside a Java servlet container. [13, 14, 15, 16, 17] 
* Servlet Container / Web Server (Apache Tomcat): A specialized environment that interprets Java code, routes web paths to Java Controllers, and executes server-side UI rendering. [18, 19] 
* Upstream Cluster: The collection or pool of backend target instances that Nginx treats as a unified group to handle web traffic. [20, 21] 
* Port Mapping / Allocation: Explicitly separating system communication channels to allow multiple network services to execute simultaneously on the same operating system.

## 🎬 How It Works Behind the Scenes: From Start to End
Here is exactly what happens when you type http://localhost/inventory-system/ into your browser window:

[User Browser] ---> (Port 80) ---> [Nginx Load Balancer]
                                             |
                     +-----------------------+-----------------------+

                     |                       |                       |
            (Port 8092)             (Port 8091)             (Port 8093)
                     v                       v                       v
             [Tomcat Server 1]       [Tomcat Server 2]       [Tomcat Server 3]


   1. The Request Landing: Your web browser sends an HTTP request to localhost. By default, browsers look for port 80.
   2. Nginx Interception: Nginx is running as a background service listening directly on port 80. It catches the request.
   3. The Load Balancing Algorithm: Nginx looks at your upstream inventory_cluster block. By default, it uses an algorithm called Round Robin. This means it loops through your servers in order: Request 1 goes to Server 1, Request 2 goes to Server 2, Request 3 goes to Server 3, and then it loops back to Server 1.
   4. Proxying Header Mapping: Nginx uses your proxy_set_header lines to forward the request to the chosen Tomcat port (e.g., 8091). It passes along the user's real IP address so Tomcat knows where the request originally came from.
   5. Tomcat Processing: Tomcat accepts the request on its HTTP connector port. It passes the path to your Spring MVC InventoryController.java.
   6. Java Execution: The Controller executes, populates your custom list of laptops, mice, and monitors, attaches your server ID visual label, and binds it to your dashboard.jsp template.
   7. The Return Journey: Tomcat renders the template into clean, standard HTML and sends it back to Nginx. Nginx takes that package and sends it right back to the user's browser screen. [22, 23, 24, 25, 26]

## 🏢 How This Works in the Real World
In a real company like Amazon or Google, this exact setup is used, but scaled up: [27, 28] 

* Instead of 3 Tomcat instances on one computer, they run hundreds of Tomcats across different virtual servers or clouds worldwide.
* Instead of a local IP (127.0.0.1), Nginx points to the private network IP addresses of those distributed cloud servers.
* They use advanced Nginx rules to handle secure encrypted connections (HTTPS/SSL), compress large web images on the fly, and cache static pages so the backend Java servers don't have to work as hard. [29, 30]
